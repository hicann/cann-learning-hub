# CANNLab 常用环境持久化配置指南

由于 CANNLab 实例仅对 `/mnt/workspace` 目录提供持久化存储，默认情况下，通过 `pip`、`npm` 安装的依赖或通过 `curl` 下载的工具，均会在实例重启后被清空。本文档提供了关于 Python 依赖、Node.js 环境及 opencode 工具的持久化配置方案。若仅使用 pypto 框架，无需安装额外依赖，可跳过本章节内容。

---

## Python 依赖管理

直接通过 `pip` 将依赖安装到默认 Python 环境中无法实现持久化。为避免实例重启后反复手动安装，提升开发效率，提供以下 2 种解决方案。

### 1. 使用脚本文件自动化重装（适用于轻量级依赖）

实例默认的 Python 环境已预装了大量常用依赖。若仅需额外补充少部分轻量级依赖，可通过编写初始化脚本，在每次重启 CANNLab 后自动执行安装。

以 `tiktoken` 分词库为例，可在 `/mnt/workspace` 目录下创建一个名为 `i-pto` 的初始化脚本：
```bash
cd /mnt/workspace
touch i-pto
```

在其中写入以下内容：

```bash
#!/bin/bash
pip install tiktoken
```

若已经在 [1.2 节](./01.02_CANNLab_pypto_quick_start.md) 中创建了 `i-pto` 脚本并配置了 pypto 的安装指令，那么该文件的完整内容应类似于：

```bash
#!/bin/bash
export TILE_FWK_DEVICE_ID=0
python3 -m pip install /mnt/workspace/dev/pypto --verbose
pip install tiktoken
```

每次重启 CANNLab 时，只需在 `/mnt/workspace` 目录执行该脚本，即可自动恢复环境：

```bash
source i-pto
```

---

### 2. 构建持久化虚拟环境（适用于复杂项目开发）

通过在 `/mnt/workspace` 目录下安装 Miniconda 或 Anaconda 等环境管理工具，可以创建完全独立的持久化虚拟环境。

以 Miniconda 为例，执行以下命令即可快速将其安装至 `/mnt/workspace/miniconda3`：

```bash
cd /mnt/workspace
wget https://repo.anaconda.com/miniconda/Miniconda3-latest-Linux-aarch64.sh -O miniconda.sh
bash miniconda.sh -b -p ./miniconda3
```

> **提示**：可前往 [Miniconda 官方仓库](https://repo.anaconda.com/miniconda/) 获取其他版本。若实例内网络不稳定，建议先下载至本地，再手动上传至 CANNLab 实例中。

安装完成后，需将 Miniconda 的可执行文件路径追加到系统的 `PATH` 环境变量中，以便直接调用 `conda` 命令：

```bash
export PATH=/mnt/workspace/miniconda3/bin:$PATH
```

验证安装是否成功：

```bash
conda --version
```

> **注意**：启用全新的 conda 虚拟环境后，实例默认环境预装的系统级依赖（如 `torch`、`torch_npu` 等）将不可见，需在虚拟环境中重新安装。此外，上述配置 `PATH` 的操作同样可以使用前文所提及的脚本文件以实现便捷加载。

## Node.js 环境管理

实例默认的 Node.js 安装路径位于：

```bash
/home/developer/.nvm/versions/node/v18.19.1/bin/node
```

通过默认 `nvm` 安装的 Node.js 版本或通过全局 `npm` 安装的包，在重启后均无法保留。由于 Node.js 环境自身较为轻量，推荐直接在 `/mnt/workspace` 下构建自定义的 Node.js 环境。

### 安装 Node.js

以 `node-v24.15.0` 版本为例，执行以下命令将其安装至 `/mnt/workspace/node/node24` 目录：

```bash
mkdir -p /mnt/workspace/node
cd /mnt/workspace/node
wget https://nodejs.org/dist/latest-v24.x/node-v24.15.0-linux-arm64.tar.xz -O node.tar.xz
tar -xvf node.tar.xz
mv node-v24.15.0-linux-arm64/ node24
rm -rf node.tar.xz

```

> **提示**：可前往 [Node.js 官方发布页](https://nodejs.org/dist/) 获取其他版本。

同样地，将自定义的 Node.js 路径加入 `PATH` 环境变量：

```bash
export PATH=/mnt/workspace/node/node24/bin:$PATH
```

验证安装结果：

```bash
node --version
```

## opencode 环境配置

opencode 官方文档参考：[https://opencode.ai/docs/zh-cn/](https://opencode.ai/docs/zh-cn/)

由于实例预装的 opencode 版本可能较旧，建议基于上文配置的自定义 Node.js 环境重新安装最新版本。

### 安装 opencode

在安装前，请确认当前使用的 `node` 命令已正确指向 workspace 目录下的持久化环境：

```bash
which node
```

预期输出应类似于：

```bash
/mnt/workspace/node/node24/bin/node
```

> **提示**：如果输出的仍是 `/home/developer` 开头的默认路径，请先执行 `export PATH=/mnt/workspace/node/node24/bin:$PATH` 更新环境变量。

通过 `npm` 全局安装 opencode：

```bash
npm install -g opencode-ai
```

验证安装：

```bash
opencode --version
```

---

### 配置 opencode 目录持久化

opencode 的运行依赖两个核心的数据与配置目录，默认位于：

```bash
~/.config/opencode
~/.local/share/opencode
```

> **说明**： `config` 目录用于存放 opencode 的全局配置（如 MCP、skill、plugin 等）。`local` 目录用于存放日志数据，以及执行 `/connect` 后连接到的 provider 鉴权信息。

为了防止重启导致配置丢失，我们需要在持久化目录中重建这两个文件夹：

```bash
mkdir -p /mnt/workspace/.config/opencode
mkdir -p /mnt/workspace/.local/share/opencode
```

通过设置环境变量，将 opencode 配置文件路径重定向至持久化目录：

```bash
export OPENCODE_CONFIG_DIR=/mnt/workspace/.config/opencode
export XDG_DATA_HOME=/mnt/workspace/.local/share
```

> **拓展**：如需从其他设备迁移已有配置，可直接将文件（如 `opencode.json` 和 `auth.json`）分别上传至上述新建的持久化目录中。

为方便日常开发，建议将环境变量的声明合并至 `i-pto` 初始化脚本中。整合后的 `i-pto` 脚本参考如下：

```bash
#!/bin/bash
# 1. 配置 pypto
export TILE_FWK_DEVICE_ID=0
python3 -m pip install /mnt/workspace/dev/pypto --verbose

# 2. 注入 Node.js 环境变量 (请替换为实际路径)
export PATH=/mnt/workspace/node/node24/bin:$PATH

# 3. 重定向 opencode 配置与数据目录
export OPENCODE_CONFIG_DIR=/mnt/workspace/.config/opencode
export XDG_DATA_HOME=/mnt/workspace/.local/share

```

后续每次重启 CANNLab 实例，只需执行 `source i-pto`，即可完成全部前置准备工作。
